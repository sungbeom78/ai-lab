## 1. 공개 원칙 (정보 공개 기준표)

| 카테고리 | 공개 수준 | 판단 기준 |
|----------|-----------|-----------|
| 시스템 아키텍처 개요 | ✅ 공개 | "어떤 구조로 만들었는가" |
| 기술 스택 및 선택 이유 | ✅ 공개 | 기술 결정의 배경과 트레이드오프 |
| 알고리즘 개념·접근 방식 | ✅ 공개 | "어떤 방식으로 접근했는가" |
| 개발 여정·변경 이력 | ✅ 공개 | 다른 개발자에게 가장 유익한 콘텐츠 |
| 확정된 거래 수익/손익 | ✅ 공개 | 투명성, 동기부여 목적 |
| 전략 파라미터·수치 | ❌ 비공개 | 핵심 IP, 모방 가능성 |
| 환경변수 키 이름 | ❌ 비공개 | 공격 벡터 힌트 |
| 디렉토리 구조 상세 | ❌ 비공개 | 보안 경로 정보 |
| 포트 번호 | ❌ 비공개 | 스캐닝 힌트 |
| 개발·운영 매뉴얼 | ❌ 비공개 | 내부 운영 절차 |
| 현재 보유 포지션·예정 종목 | ❌ 비공개 | 전략 노출, front-running 위험 |
| 미확정(보유 중) 손익 | ❌ 비공개 | 위와 동일 |

### 수익 공개 기준 — 상세

공개 대상은 **매도로 손익이 확정된 거래**만입니다.

```
공개 OK  : 체결 완료 → 청산 완료 → 손익 확정된 거래
공개 금지 : 현재 매수 중인 포지션
공개 금지 : 매수 예정 또는 시그널 발생 종목
공개 금지 : 미실현 손익 (보유 중 평가손익)
```

대시보드의 실시간 "활성 거래" 섹션은 **Zero Trust 접근 제한 후에만** 노출합니다.
그 전까지는 해당 섹션을 숨기거나 빈 상태로 유지합니다.

### 2-2. `/docs/system_config` — 시스템 설정

**삭제 또는 수정할 내용**

| 항목 | 조치 |
|------|------|
| 서버 하드웨어 명칭 (G3 Mini PC 등) | 추상화 → "메인 서버 / DB 서버" |
| 프로젝트 디렉토리 구조 전체 | 삭제 → 레이어 흐름 설명으로 교체 |
| 환경변수 키 목록 테이블 | 삭제 → 카테고리 설명으로 교체 |
| 리스크 파라미터 수치 컬럼 | 수치만 삭제, 파라미터 이름·설명 유지 |
| 오케스트레이터 파라미터 (포트 포함) | 전체 삭제 |
| DB/Redis 환경변수명, Pool Size | 삭제 → "PostgreSQL + Redis 조합" 만 유지 |

**교체 문구 예시**

```markdown
# 디렉토리 구조 대신:
소스 코드는 역할별로 레이어가 분리되어 있습니다.
주요 흐름: 어댑터 → 수집기 → 선정기 → 시그널 → 리스크 → 실행

# 환경변수 대신:
환경변수는 브로커 API 인증, DB 접속, 알림 서비스 연동에 사용됩니다.
민감정보는 .env 파일로만 관리하며 버전 관리에 포함하지 않습니다.

# 리스크 파라미터 수치 대신:
| 파라미터 | 설명 |
|----------|------|
| max_open_positions | 동시에 유지할 수 있는 최대 포지션 수 |
| max_daily_loss_pct | 일일 손실 한도 초과 시 당일 거래 중단 |
| stop_loss | 기본 손절 기준 (hard stop 포함) |
| trailing_stop | 수익 보호를 위한 트레일링 스톱 |
```

**유지할 내용**

- 기술 스택 (PostgreSQL, Redis, Python, Windows + WSL 구성 및 이유)
- 활성 마켓 테이블 (마켓 종류, 어댑터, 세션 시간)

### 2-4. `/docs/trading_logic` — 거래 로직

가장 민감한 문서입니다. **개념·구조는 공개, 수치·임계값은 전면 제거**합니다.

**삭제 대상 — 수치 전체**

| 항목 | 조치 |
|------|------|
| 필터 단계별 가중치 % | 컬럼 자체 제거 |
| 캔들 스코어링 임계값·점수 | 개념 설명으로 교체 |
| 진입 로직 봉 수, 스코어 | N봉으로 추상화, 스코어 컬럼 제거 |
| EMA 민감도 임계값 수치 | 삭제 |
| 청산 로직 pip 수치 | "일정 pips" 로 추상화 |
| R:R 비율, 승률, PF, MDD 수치 | 전체 삭제 |
| 한국주식 스코어링 가중치 | 수치 제거, 요소 이름만 유지 |
| 한국주식 진입·청산 조건 수치 | 전체 제거 |
| 공통 리스크 관리 수치 | 수치 컬럼만 제거 |
| 스크립트 실행 예시·결과 샘플 | 전체 삭제 (실제 스코어 노출됨) |

**교체 문구 예시**

```markdown
# 진입 로직 변경 전:
| Breakout Long | LONG | 현재 종가 > 직전 8봉 고가 + EMA 상승 추세 | 75 |

# 변경 후:
| Breakout Long | LONG | 최근 N봉 고가 돌파 + EMA 추세 확인 |

# 캔들 스코어링 변경 전:
아랫꼬리 평균 > 0.3 → +20점

# 변경 후:
아랫꼬리 비율을 분석해 하방 지지력 지표로 활용
```

**유지할 내용 (개념 수준)**

- 4단계 필터링 파이프라인 구조 (단계 이름, 개념)
- 캔들 품질 분석 접근법
- 비상관 종목 선택 철학 (국내주식 집중 vs 외환 분산)
- Breakout / Pullback 패턴 이름
- EMA Cross 필터 사용 사실
- 손절·익절·트레일링·시간청산 조합 구조
- 킬스위치(`KILL_SWITCH` 파일) 메커니즘

## 3. 수익 공개 페이지 설계

### 표시할 데이터

```
확정 거래 목록 (페이지네이션)
  - 청산 일시
  - 마켓 (KR_STOCK / FX_FUT 등)
  - 손익 금액 / 손익률
  - 보유 시간

요약 통계
  - 이번 달 확정 손익
  - 누적 확정 손익
  - 총 거래 건수
  - 승률 (확정 기준)
  - 월별 손익 차트
```

### 표시하지 않을 데이터

```
- 종목명 (어떤 종목을 거래했는지)
- 진입 가격, 청산 가격
- 현재 보유 중인 포지션
- 미실현 손익
- 보유 수량
```

> 종목명을 숨기는 이유: 어떤 종목을 언제 거래하는지 패턴이 노출되면
> 전략이 역산될 수 있습니다. 손익만 공개해도 투명성은 충분합니다.

## 4. 업데이트 운영 규칙

### 업데이트 주기

- **매주 1회** (요일 고정 권장, 예: 매주 일요일)
- 수동 업데이트: 문서 내용 수정 후 웹서버에 반영
- 수익 데이터: 자동 집계 (DB → 대시보드 실시간 또는 배치)

### 업데이트 대상별 분류

| 대상 | 주기 | 방식 |
|------|------|------|
| 확정 수익 통계 | 실시간 또는 일 1회 | 자동 (DB 집계) |
| 거래 로직 문서 | 전략 변경 시에만 | 수동 |
| 시스템 설정 문서 | 아키텍처 변경 시에만 | 수동 |
| 개발 여정 / 변경 이력 | 주 1회 | 수동 (주간 업데이트) |
| 기술 스택 / 소개 문구 | 변경 시에만 | 수동 |

### 주간 업데이트 체크리스트

매주 업데이트 시 아래 항목을 점검합니다.

```
[ ] 이번 주 주요 변경사항 change_log에 기록했는가
[ ] 새로 추가한 기능·모듈이 있으면 system_impl 반영했는가
[ ] 전략 파라미터 변경이 있었다면 trading_logic 수치가 노출되지 않는지 재확인
[ ] 수익 데이터가 확정 거래만 표시되는지 확인
[ ] 보안 민감 정보가 새로 노출된 내용은 없는지 확인
[ ] dev_manual / ops_manual이 공개 경로로 접근되지 않는지 확인
```

### 문서 작성 규칙

공개 문서를 새로 작성하거나 수정할 때 적용할 규칙입니다.

```
1. 수치가 들어가면 일단 비공개로 간주하고 검토
2. 파일 경로가 들어가면 `config/`, `ssl/`, `cookies/` 포함 여부 확인
3. 환경변수명이 나오면 제거
4. 포트 번호가 나오면 제거
5. 코드 예시에 실제 스코어·결과가 보이면 제거
6. "이 문서는 자동 생성됩니다"류 주석도 확인 — 생성 스크립트 경로 노출 주의
```

## 5. 추가하면 좋을 콘텐츠 (현재 없는 것)

지금 문서는 내부 설명서 느낌입니다. 외부 개발자가 흥미를 느끼려면 아래 콘텐츠가 있어야 합니다.

### 5-1. 기술 결정 배경 페이지

"왜 이 기술을 선택했는가"는 다른 개발자에게 가장 유익한 콘텐츠입니다.

```markdown
예시:
- 왜 PostgreSQL인가? → 시계열 캔들 + 구조적 거래 데이터를 함께 다루기 위해
- 왜 MT5인가? → 해외선물 접근성이 가장 좋은 플랫폼
- 왜 Windows + WSL인가? → MT5가 Windows 전용이라 WSL로 Python 환경 분리
- 왜 Redis인가? → 실시간 로그 스트림과 캐시를 가볍게 처리하기 위해
```

### 5-2. 개발 여정 / 변경 이력 공개

`doc/change_log_archive/`의 내용을 요약해 공개하는 것이 좋습니다.

```markdown
공개하면 좋은 내용:
- v1 → v2 → v3 구조 변화와 이유
- 어떤 문제를 만나서 어떻게 해결했는지
- 실패한 접근법과 그 이유
- 백테스트에서 발견한 엣지케이스
```

이런 내용이 개발 커뮤니티에서 가장 높은 반응을 얻습니다.

### 5-3. 프로젝트 소개 문구 (대시보드 상단)

```markdown
Bomiyang's Trade System은 개인 자동매매 시스템입니다.
해외선물·국내주식 등 멀티 마켓 알고리즘 트레이딩을 구현하고 있습니다.

재미 90%, 개인투자 10%의 사이드 프로젝트입니다.
알고리즘 개념과 개발 여정은 공개하며, 확정된 수익도 투명하게 공개합니다.
전략의 세부 파라미터는 공개하지 않습니다.
```

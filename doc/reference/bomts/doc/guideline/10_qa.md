# BOM_TS -- QA 실행 및 운영 지침

> Version: 1.0 | Created: 2026-03-28

---

## 1. 절대 규칙

- **모든 QA 실행은 반드시 결과를 텔레그램으로 전송한다**
- 텔레그램 전송 실패 시: 3회 재시도(3초 간격) -> 실패 시 WARNING + `tmp/qa/telegram_error.log` 기록
- QA 결과 자체(status)는 텔레그램 전송 성패와 무관하다

---

## 2. QA 레벨

| 레벨 | 목적 | 검사 항목 |
|------|------|----------|
| qa_quick | 개발 중 즉시 검증 | SoT + data flow (grep) + changed-file identifier audit |
| qa_standard | 운영 전 검증 | Quick + runtime consistency |
| qa_full | 전체 정밀 점검 | Standard + AST + API checks + identifier audit |

---

## 3. 실행 시점

| 시점 | 레벨 | 트리거 |
|------|------|--------|
| AI 작업 직후 / phase 단위 개발 완료 시 | qa_quick | 수동 |
| 월~금, 일요일 07:30 (시스템 초기화 직후) | qa_standard | 스케줄러 |
| 커밋 직전 | qa_standard | 수동 (필수) |
| 매주 토요일 07:30 | qa_full | 스케줄러 |

---

## 4. 실행 명령

```bash
python run_test.py qa_quick
python run_test.py qa_standard
python run_test.py qa_full
python script/operation/audit_identifiers.py
```

### qa_full 추가 점검 (용어 감사)

- `qa_full`에는 `python script/operation/audit_identifiers.py`를 포함한다.
- 미등록 단어 발견 시 결과는 `WARN`으로 처리한다 (`FAIL` 아님).

### qa_quick 추가 점검 (변경 파일 대상 감사)

- `qa_quick` targeted identifier audit 타겟 우선순위:
- 1) 변경된 `src/**/*.py`, `script/db/*.sql`
- 2) `QA_IDENTIFIER_TARGETS` 환경변수(쉼표/개행 구분 상대 경로 목록)
- 3) 운영 기본 profile fallback (메시지/실행 핵심 파일군)
- 감사 결과에서 `FATAL`, `ERROR`는 `FAIL`로 처리한다.
- 그 외(`WARN` 등)는 audit artifact와 executed check detail에 남기되, `qa_quick` overall status를 강등하지 않는다.

### audit_identifiers.py 사용 예시

```bash
python script/operation/audit_identifiers.py --changed-only
python script/operation/audit_identifiers.py --changed-only online
python script/operation/audit_identifiers.py --files src/common/config.py script/db/schema.sql
python script/operation/audit_identifiers.py --dir src/execution
python script/operation/audit_identifiers.py --tables PG_DSN stock_master trade_history
python script/operation/audit_identifiers.py --db-env PG_DSN
```

## 5. 결과 해석

### status

| 값 | 의미 | 조치 |
|----|------|------|
| PASS | 정상 | 진행 가능 |
| WARN | 경고 | 확인 필요 |
| FAIL | 실패 | 커밋 금지, 운영 중단 |

### confidence

| 값 | 의미 | 조치 |
|----|------|------|
| FULL | 모든 체크 실행 | 신뢰 가능 |
| PARTIAL | 일부 SKIP | 확인 필요 |
| LOW | 핵심 체크 SKIP | qa_standard/qa_full일 경우 **운영 금지** |

---

## 6. FAIL 대응 규칙

1. 해당 변경 **커밋 금지**
2. 운영 중일 경우: 해당 마켓 트레이딩 **즉시 중단** (2개 이상 마켓이면 전체 중단)
3. 원인 해결 전 **재실행 금지**
4. CRITICAL > 0 -> 운영 절대 금지

---

## 7. LOW confidence 대응 규칙

- qa_standard 또는 qa_full에서 LOW PASS라도 **운영 금지**
- 운영 중이면 전체 마켓 트레이딩 중단

---

## 8. 결과 파일

| 파일 | 용도 |
|------|------|
| `tmp/qa/latest.json` | 최신 결과 |
| `tmp/qa/latest_quick.json` | 최신 Quick 결과 |
| `tmp/qa/latest_standard.json` | 최신 Standard 결과 |
| `tmp/qa/latest_full.json` | 최신 Full 결과 |
| `tmp/qa/last_success.json` | 마지막 PASS 시각 + level |
| `tmp/qa/archive/{timestamp}.json` | 아카이브 (30일 자동 삭제) |
| `tmp/qa/running.lock` | 실행 중 잠금 |
| `tmp/qa/telegram_error.log` | 텔레그램 전송 실패 로그 |

### 최종 보고 연계

- 최종 보고 생성기는 `tmp/qa/latest_{level}.json`과 `tmp/qa/archive/{timestamp}.json`을 읽어 level별 수행 여부와 결과를 수집한다.
- 따라서 QA 결과 JSON에는 `command`, `executed_checks`, `check_summary`, `artifact`, `result_summary`가 포함되어야 한다.
